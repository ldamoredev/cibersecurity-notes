Cross-Site Scripting (XSS)

Definition

XSS happens when untrusted input is included in a web page or browser-executed context in a way that allows attacker-controlled JavaScript to run. It can be reflected, stored, or DOM-based depending on where the unsafe handling occurs.

Why it matters

XSS is not just “alert(1)”. It turns a browser into an execution environment under attacker influence. That can lead to session theft, action forgery, UI redressing, data exfiltration, and pivoting into other application functionality.

Attacker perspective

Attackers test: - reflected inputs - stored content fields - rich text or markdown features - query parameters rendered into the DOM - JavaScript sinks and templating mistakes

They care about context: HTML, attribute, script, URL, CSS, or DOM sink.

Defender perspective

Defenders should: - output-encode by context - sanitize only where appropriate and carefully - avoid unsafe DOM sinks - use CSP as a mitigation layer, not a substitute for correct handling - assume XSS can amplify the impact of many other bugs

Practical examples

  • A search term is reflected unsafely into HTML.
  • A comment field stores attacker input that every admin later renders.
  • A frontend script takes a query parameter and injects it directly into innerHTML.
