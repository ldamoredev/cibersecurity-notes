IDOR

Definition

Insecure Direct Object Reference is a web-access pattern where an attacker changes an object identifier in a request and the application returns or modifies another user's data because the server does not enforce object-level authorization correctly.

Why It Matters

IDOR is one of the clearest ways broken access control becomes obvious in real applications. It frequently appears in browser-backed CRUD flows, download routes, profile pages, and other object-centric web features.

This note is the web-facing object-reference pattern. Broken Object Level Authorization is the API-side note for the same underlying failure when object authorization is expressed through API resources and tenant-scoped data models.

Attacker Perspective

Attackers capture a valid request, change only the object reference, and look for unauthorized reads, writes, or existence leaks. They often test predictable IDs, hidden links, export routes, and object references exposed in forms or client-side code.

Defender Perspective

Defenders should treat every object lookup as an authorization decision, not just a data fetch. UI visibility, obfuscated identifiers, and route hiding do not replace server-side ownership or policy checks.

Practical Examples

  • Changing /account/123 to /account/124 reveals another user's profile.
  • A download endpoint accepts invoiceId=9321 and returns another customer's PDF.
  • A frontend hides edit controls, but replaying the underlying request still updates another user's record.
