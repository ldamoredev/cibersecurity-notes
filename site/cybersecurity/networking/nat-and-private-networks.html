NAT and Private Networks

Definition

Private networks use non-public IP ranges internally, while NAT (Network Address Translation) lets those hosts communicate outward or appear behind a smaller set of public addresses. This creates a separation between public reachability and internal addressing.

Why it matters

This note matters because many modern attacks depend on the difference between: - what is reachable from the Internet - what is reachable from inside the application environment

This is why SSRF, cloud metadata access, and lateral movement can be much more serious than they first appear.

Attacker perspective

Attackers look for ways to make a public-facing system talk to private addresses on their behalf. That is one reason SSRF is dangerous: the app may be able to access internal hosts or metadata endpoints even when the attacker cannot directly do so.

Defender perspective

Defenders need to understand: - which services are internal-only - which egress and internal routing paths exist - how applications can reach metadata, admin APIs, or databases - how NAT and private subnets can create a false sense of safety if app-level bugs still allow internal access

Practical examples

  • A server-side image fetcher can request http://169.254.169.254 in some environments.
  • A backend in a private subnet can still access internal APIs that external users cannot see.
  • A database is not public, but a compromised app server can reach it directly.

Useful commands

# Inspect local routes / interfaces
ip route
ip addr

# Test internal reachability from a controlled host
nc -vz 10.0.0.5 8080
